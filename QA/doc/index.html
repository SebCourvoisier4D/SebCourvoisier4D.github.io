Wakanda QA Internal Tools

Purpose

The aim is to develop a single package of node.js modules that provides all the tools needed by the QA team to run the tests and report their results (either programmatically or using the command line).

bin/clean.js

(STABLE) Clean all the folders used by the tests:

  • The Temp folder: the folder used to download and extract the builds, as defined by the tempDir property of the configuration file
  • The Builds folder: the folder used to store the builds, as defined by the buildsDir property of the configuration file
  • The Test folder: the folder used to execute the tests, as defined by the testDir property of the configuration file
  • The Results folder: the folder used to store the results of the tests, as defined by the resultsDir property of the configuration file

Usage: qa-clean [<buildsToKeep>]

  • buildsToKeep (number, optional): The number of builds to keep (for both the Results and Builds folders). It uses the buildsToKeep property of the configuration file by default.

Note: the alias qa-clean is made available using the npm link command.

bin/download.js

(STABLE) Download the latest Wakanda build available according to TeamCity.

Usage: qa-download [<branch>] [--with-4d] [--with-installer] [--with-shell]

  • branch (string, optional): the Wakanda branch to download. It uses the wakandaBranch property of the configuration file by default.
  • --with-4d (optional): If set, also gets the latest 4D Server (relates to the Wakanda branch).
  • --with-installer (optional): If set, also gets the Wakanda installer.
  • --with-shell (optional): If set, also gets the Wakanda Shell (Linux and Mac only).

Note: the alias qa-download is made available using the npm link command.

bin/init.js

(WORK IN PROGRESS) Create a new configuration file (this script will be used to finalize the installation of the QA tools).

Usage: qa-init

Prerequisites (some of those could be automatically installed in the future):

  • Node.js and Git (with the command line tool).
  • P4 Client (with the command line tool and a configured workspace).
  • Google Chrome.
  • Java SDK.

Note: the alias qa-init is made available using the npm link command.

bin/sync-all.js

(STABLE) Sync all the tests on the local P4 workspace.

Usage: qa-sync-all [--force]

  • --force (optional): If set, performs a forced syncing.

Note: the alias qa-sync-all is made available using the npm link command.

bin/sync-test.js

(STABLE) Sync the given test on the local P4 workspace.

Usage: qa-sync-test <test_path> [--force]

  • test_path (string): Full path of the test to sync (path to its .json or .xml configuration file in the local P4 workspace).
  • --force (optional): If set, performs a forced syncing.

Note: the alias qa-sync-test is made available using the npm link command.

qa-scripts/csjs-headless.js

(STABLE) Run the given test using PhantomJS/SlimerJS/TrifleJS. It means that this script has to be run using one of these headless tools, for instance:

phantomjs /path/to/qa-scripts/csjs-headless.js <test_URL> <report_path>

The script expects 2 arguments:

  • test_URL: The URL of the test to run (e.g. http://127.0.0.1:8081/index.waPage/index.html?waktest-path=/full/path/to/the/test.js&waktest-format=JUnit)
  • report_path: The path where the report will be created (as report.json, report.xml or report.html according to the given waktest-format URL parameter)

qa-scripts/studio-monitor.js

(STABLE) This script is meant to be launched by the UnitTest extension of Wakanda Studio when running in test mode. It instantiates a WebSocket server that acts as a relay between the UnitTest extension and the module:BuildRunner module. It allows the test agent to send commands to the Studio and receive its events in real-time.
